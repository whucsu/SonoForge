# Этап 2: Финальная архитектурная спецификация (Software Design Document)
**Проект:** Персональный эхокардиографический аналитический инструмент  
**Статус:** Утверждено для перехода к Спринту 1 (Proof of Concept & Skeleton)  
**Архитектурный вектор:** Локальный десктоп, клиническая терминология (ASE/EACVI), CPU-инференс (ONNX), работа с локальными папками.

---

## 📌 Executive Summary
Данный документ фиксирует инженерный фундамент для разработки персонального инструмента. Проект **не является** медицинским изделием (SaMD) или исследовательским инструментом (RUO). Из архитектуры исключены водяные знаки, дисклеймеры, веб-интеграции (DICOMweb) и любые модули деидентификации — приложение работает с данными «как экспортированы» с УЗ-аппарата.

Архитектура построена на Clean Architecture с чётким разделением ответственности. Рендеринг — **единый стек PyQtGraph** (без `QGraphicsView`). DICOM-парсинг изолирован в Infrastructure; Domain оперирует только dataclass-моделями. ONNX-инференс — в Infrastructure; пред/постобработка маски — в Domain.

## 1. Интегрированный технологический стек

| Компонент | Технология | Обоснование выбора и ограничения |
| :--- | :--- |--- |
| **Управление зависимостями** | `uv` + `pyproject.toml` | Полный отказ от `requirements.txt`. `uv.lock` — воспроизводимые сборки. |
| **GUI Framework** | `PySide6` | LGPL, стабильный десктопный стек. |
| **Визуализация** | `PyQtGraph` (единый стек) | `GraphicsLayoutWidget` + `ViewBox`: `ImageItem` (пиксели), `PlotDataItem` / `ScatterPlotItem` / `ROI` (оверлеи). `autoDownsample` обязателен. **QGraphicsView не используется.** |
| **DICOM Парсинг** | `pydicom` + `pylibjpeg` + `pylibjpeg-openjpeg` + `pylibjpeg-libjpeg` | `python-gdcm` **запрещён**. Покрытие JPEG 2000, RLE (GE, Philips, Siemens). |
| **Обработка изображений** | `OpenCV` (`cv2`) | Декодирование MP4, предпроцессинг кадров для ONNX (resize, normalization). Только Infrastructure. |
| **Математика и ИИ** | `NumPy`, `SciPy`, `onnxruntime` (CPU) | Расчёты в Domain. `onnxruntime` — только Infrastructure. `numpy>=1.26,<2.0`. |
| **Многопоточность (I/O)** | `QRunnable`, `QThreadPool` | Фоновое чтение DICOM без блокировки Main Thread. |
| **Многопоточность (Math/AI)** | `ProcessPoolExecutor` | ONNX-инференс в отдельном процессе (обход GIL). |
| **Генерация отчётов** | `ReportLab` | PDF с кириллицей. Модуль в Application Layer. |

## 2. Четырехуровневая слоистая архитектура (Clean Architecture)

Зависимости направлены строго внутрь: **Presentation → Application → Domain ← Infrastructure**

### 2.1. Слой представления (Presentation Layer)

* **Ответственность:** Отрисовка UI, ввод (мышь, клавиатура, hotkeys).
* **Компоненты:**
  * `MainWindow` — layout, панели, переключение режимов 2D / Doppler / Side-by-side (Фаза 2).
  * `ViewerWidget` — `GraphicsLayoutWidget` + `ViewBox` + `ImageItem` + оверлеи (`PlotDataItem`, `LineROI`).
  * `DopplerWidget` — `PlotWidget` для спектрограммы; маркеры (`ScatterPlotItem`), трассировка VTI (`PlotDataItem`).
  * `LocalBrowserWidget` — дерево исследований.
  * `MeasurementPanel` — правая панель результатов.
* **Ограничения:** Запрещены импорты `pydicom`, `cv2`, `onnxruntime`. Работа только с `QImage`, координатами и domain-DTO.
* **Терминология:** LVEF, КДО, КСО, E, A, e', E/e', VTI, DT, IVRT, AT, PGpeak, PGmean.

### 2.2. Слой приложения (Application Layer)

* **Ответственность:** Координация, состояние, оркестрация workers.
* **Компоненты:**
  * `AppController` — точка входа use-case'ов (открыть папку, загрузить серию, измерить).
  * `MasterClock` *(Фаза 2)* — глобальное `current_time_ms` для side-by-side синхронизации.
  * `StateManager` — кэш расчётов, текущее исследование, активные маркеры.
  * `ReportService` — сбор данных и вызов PDF/CSV/JSON *(Фаза 2)*.
  * `Workers` — `DicomLoaderWorker`, `OnnxWorker` (`QRunnable` / `ProcessPoolExecutor`).

### 2.3. Слой домена (Domain Layer)

* **Ответственность:** Бизнес-логика. **Чистый Python** — без Qt, pydicom, cv2, onnxruntime, ФС.
* **Компоненты:**
  * `domain/models/` — dataclass-модели (см. §5).
  * `domain/calculations/` — `lvef_simpson.py`, `teichholz.py`, `doppler_metrics.py`, `bernoulli.py`, `chamber_volumes.py`, `rv_function.py`, `diastolic_function.py`.
  * `domain/services/segmentation_service.py` — **только** `prepare_tensor()`, `mask_to_contour()`, `smooth_contour()` (NumPy). Без загрузки модели.
* **Запрещено в Domain:** `pydicom.Dataset`, `onnxruntime`, `QImage`.

### 2.4. Инфраструктурный слой (Infrastructure Layer)

* **Ответственность:** ФС, кодеки, DICOM, ONNX runtime.
* **Компоненты:**
  * `LocalDicomDirectoryScanner` — рекурсивный сканер; дерево `Дата/Время → Серия → Экземпляр`.
  * `DicomMetadataMapper` — читает `pydicom.Dataset`, **маппит** в domain `StudyMetadata` / `SeriesMetadata` / `InstanceMetadata`.
  * `DicomReaderImpl` — декомпрессия пикселей → `numpy.ndarray`.
  * `VideoReader` — MP4 через OpenCV, Ring Buffer (50 кадров).
  * `OnnxInferenceEngine` — `InferenceSession`, `run()` → сырой тензор маски. Вызывает Domain `segmentation_service` для постобработки.
  * `EcgWaveformParser` *(Фаза 2)* — извлечение R-зубца из DICOM Waveform Sequence.

## 3. ONNX: модель, слои и оптимизация памяти

### 3.1. Утверждённая модель: EchoNet Segmentation Lite

| Параметр | Значение |
|---|---|
| **Источник** | [echonet/dynamic](https://github.com/echonet/dynamic), MIT |
| **Базовая** | DeepLabV3-ResNet50 (`deeplabv3_resnet50_random.pt` → ONNX) |
| **Lite (fallback)** | DeepLabV3-MobileNetV3-Large → ONNX |
| **Исключено** | R2+1D / R3D / MC3 (видео EF) — не загружать |

Контракт: вход `(1, 3, 112, 112)` float32; выход — маска ЛЖ; инференс **только на кадрах ED/ES**.

Полная спецификация — `Этап 1.md`, Приложение В.

### 3.2. Разделение Domain vs Infrastructure

| Операция | Слой | Модуль |
|---|---|---|
| Resize 112×112, нормализация mean/std | Domain | `segmentation_service.prepare_tensor()` |
| Загрузка `.onnx`, `session.run()` | Infrastructure | `onnx_engine.OnnxInferenceEngine` |
| Маска → контур, масштаб в координаты кадра | Domain | `segmentation_service.mask_to_contour()` |
| INT8 quant (опционально) | Infrastructure | `onnx_engine.quantize_model()` |
| Запуск в фоне | Application | `OnnxWorker` → `ProcessPoolExecutor(max_workers=1)` |

### 3.3. Анти-OOM политика

1. Один worker процесса инференса
2. Инференс на остановленном кадре (не во время play)
3. При `MemoryError` / timeout > 2 с → fallback на Lite-модель или уведомление «используйте ручной контур»
4. Порог RAM: ResNet50 при ≥ 16 GB; MobileNetV3-Lite при 8 GB

### 3.4. Файлы моделей

```text
models/
├── echonet_seg_resnet50.onnx
├── echonet_seg_mobilenet_v3.onnx   # опционально
└── model_manifest.json             # в git; .onnx — локально
```

## 4. Рендеринг: единый стек PyQtGraph

Смешение `ImageView` + `QGraphicsView` исключено — разные системы координат и событий усложняют синхронизацию при 60 FPS.

**Паттерн 2D Viewer:**
```
GraphicsLayoutWidget
└── ViewBox (invertY=True, aspectLocked=True)
    ├── ImageItem          # кадр DICOM/MP4
    ├── PlotDataItem       # контур (ручной / ИИ)
    ├── LineROI / ROI      # калиперы, линейные размеры
    └── ScatterPlotItem    # узлы сплайн-редактора (Фаза 2)
```

**Паттерн Doppler Widget:**
```
PlotWidget
├── ImageItem или PlotDataItem   # спектрограмма
├── ScatterPlotItem              # пиковые маркеры (E, A, e', Vmax)
├── PlotDataItem (2 точки)       # интервальные маркеры (DT, IVRT, AT)
└── PlotDataItem (polyline)      # ручная трассировка VTI
```

Все оверлеи живут в одном `ViewBox` / `PlotWidget` — общая система координат, один цикл отрисовки.

## 5. Domain-модели (dataclass)

```python
@dataclass(frozen=True)
class InstanceMetadata:
    sop_instance_uid: str
    series_uid: str
    modality: str               # US, OT
    number_of_frames: int
    pixel_spacing: tuple[float, float] | None
    frame_time_ms: float | None
    series_description: str

@dataclass(frozen=True)
class SeriesMetadata:
    series_uid: str
    study_uid: str
    modality: str
    description: str
    instances: list[InstanceMetadata]

@dataclass(frozen=True)
class StudyMetadata:
    study_uid: str
    study_datetime: datetime    # ключ группировки в браузере
    series: list[SeriesMetadata]

@dataclass
class DopplerPeakMarker:
    label: str                  # "E", "A", "e_sept", "Vmax", ...
    time_ms: float
    velocity_cm_s: float

@dataclass
class DopplerIntervalMarker:
    label: str                  # "DT", "IVRT", "AT"
    start_time_ms: float
    end_time_ms: float

@dataclass
class DopplerTrace:
    label: str                  # "VTI"
    points: list[tuple[float, float]]  # (time_ms, velocity_cm_s)

@dataclass
class Contour:
    phase: str                  # "ED" | "ES"
    view: str                   # "A4C" | "A2C" | ...
    points: list[tuple[float, float]]  # пиксельные координаты
    source: str                 # "manual" | "ai"
```

## 6. Порты и интерфейсы (Protocol)

```python
# domain/ports.py — абстракции, реализуемые Infrastructure

class IDicomReader(Protocol):
    def read_pixels(self, path: Path) -> np.ndarray: ...
    def read_metadata(self, path: Path) -> InstanceMetadata: ...

class IStudyScanner(Protocol):
    def scan(self, root: Path) -> list[StudyMetadata]: ...

class IOnnxSegmenter(Protocol):
    def segment(self, frame: np.ndarray) -> np.ndarray: ...  # маска H×W

# application/use_cases.py
class LoadStudyTree: ...
class MeasureDoppler: ...       # DopplerTrace + markers → DopplerResults
class CalculateLvef: ...        # Contour[] → LvefResult
```

**Поток DopplerWidget → Domain:**
1. `DopplerWidget` эмитит `markers_changed(DopplerMeasurementDTO)`.
2. `AppController` передаёт DTO в `doppler_metrics.compute(dto)`.
3. `StateManager` сохраняет `DopplerResults`, сигнал обновляет `MeasurementPanel`.

## 7. Управление памятью и передача данных

1. **Worker → GUI:** `numpy.ndarray` → `QImage` в worker-потоке → `Signal(QImage)` в Main Thread.
2. **Ring Buffer MP4:** 50 кадров, перезапись по кругу.
3. **Освобождение:** обнуление ссылок на тяжёлые массивы после отрисовки.

## 8. Обработка ошибок

| Ситуация | Поведение |
|---|---|
| Неподдерживаемый Transfer Syntax | Сообщение в статус-баре + пропуск файла при сканировании |
| Отсутствует `PixelSpacing` | Fallback: ручная калибровка по шкале глубины |
| Битый / неполный DICOM | Лог + запись в `scan_errors.log`, серия помечается ⚠ |
| ONNX timeout (>2 с) | Отмена worker, уведомление «сегментация недоступна» |
| Нет кадра ЭКГ в DICOM | ED/ES только вручную (hotkeys); ECG-парсер — Фаза 2 |
| Сетевой диск недоступен | Диалог с повтором / выбором другой папки |

## 9. Стратегия тестирования

| Уровень | Что тестируем | Инструмент |
|---|---|---|
| Unit (Domain) | Симпсон, Тей-Хольц, doppler_metrics, bernoulli | pytest + эталонные числа |
| Unit (Infrastructure) | DicomMetadataMapper на синтетических Dataset | pytest + pydicom |
| Integration | Чтение реальных DICOM → метаданные + 1 кадр | pytest, `@pytest.mark.integration` |
| Performance | Время сканирования дерева | pytest-benchmark |
| AI Validation *(Фаза 2)* | ONNX контур vs эталон | ±3% LVEF vs EchoPAC, только AI/авто |

**Эталонный набор тестовых DICOM** (`tests/fixtures/`):
- `reference_manifest.json` — реестр файлов: vendor (GE/Philips/Siemens), модальность, ожидаемые теги.
- **Tier 1 (обязательный):** 3–5 собственных экспортов с аппарата (2D, PW, TDI, CW) — хранятся локально, **не коммитятся** в git.
- **Tier 2 (опционально):** подмножество Cardiac Atlas Project для CI без PHI.
- **Tier 3:** синтетические DICOM, генерируемые скриптом `tests/fixtures/generate_synthetic_dicom.py` для unit-тестов парсера.

## 10. Критерии производительности и валидации

| Метрика | Критерий |
|---|---|
| Построение дерева | **< 3 с** — до 100 исследований / 1000 серий на локальном NVMe SSD |
| Построение дерева (сеть) | **< 8 с** — те же объёмы на gigabit SMB/NFS |
| ONNX-сегментация *(Фаза 2)* | < 500 мс на кадр A4C, CPU i7/Ryzen 7 |
| LVEF AI vs EchoPAC *(Фаза 2)* | ±3% — **только** для AI-контура и автоматических измерений |
| Ручные измерения | Клиническая точность обеспечивается врачом; формальная сверка с EchoPAC не требуется |
| PDF *(Фаза 2)* | < 3 с |
